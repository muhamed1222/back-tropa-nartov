# 🔍 Критический анализ настройки Strapi для маршрутов

## ❌ КРИТИЧЕСКИЕ ПРОБЛЕМЫ

### 1. **ПРОБЛЕМА: Порядок мест НЕ сохраняется**

**Проблема:**
- Strapi many-to-many relations **НЕ гарантируют сохранение порядка** элементов
- Порядок массива `places` может теряться при сохранении в БД
- В PostgreSQL порядок в промежуточной таблице не гарантирован

**Последствия:**
- Маршрут может быть построен в неправильном порядке
- Расстояние будет рассчитано между неправильными точками
- Редактор выберет места в порядке A→B→C, но система может сохранить B→A→C

**Решение:**
Нужна промежуточная таблица с полем `order_num` (как в Go бэкенде - `RouteStop`):
- Создать компонент `RouteStop` с полями: `place`, `order_num`
- Использовать one-to-many связь вместо many-to-many
- Или использовать кастомный плагин для ordered relations

---

### 2. **ПРОБЛЕМА: Дублирование полей**

**Проблема:**
```json
"distance": { "type": "decimal" },      // старое поле
"distance_km": { "type": "decimal" },   // новое поле
"duration": { "type": "decimal" },       // старое поле  
"duration_hours": { "type": "decimal" } // новое поле
```

**Последствия:**
- Дублирование данных в БД
- Риск рассинхронизации (одно поле обновлено, другое нет)
- Усложнение поддержки
- Путаница для разработчиков

**Решение:**
- Удалить старые поля `distance` и `duration`
- Или использовать только старые поля (для обратной совместимости)
- Мигрировать данные из старых полей в новые

---

### 3. **ПРОБЛЕМА: Нет валидации координат**

**Проблема:**
- Не проверяется, что у всех мест есть координаты перед расчетом
- Если у места нет координат, оно просто пропускается
- Нет предупреждения редактору

**Последствия:**
- Редактор может выбрать место без координат
- Расстояние будет рассчитано неправильно
- Нет обратной связи об ошибке

**Решение:**
- Добавить валидацию в lifecycle: проверять координаты перед расчетом
- Выбрасывать ошибку, если у места нет координат
- Показывать предупреждение в UI Strapi

---

### 4. **ПРОБЛЕМА: Фиксированная скорость расчета**

**Проблема:**
```javascript
const averageSpeed = 3; // км/ч - всегда одинаковая
```

**Последствия:**
- Автомобильный маршрут будет рассчитан как пеший (3 км/ч)
- Велосипедный маршрут будет рассчитан неправильно
- Нет учета сложности маршрута (горы, равнина)

**Решение:**
- Использовать `type_id` для определения скорости
- Добавить поле `average_speed` в модель Type
- Или добавить поле `speed_kmh` в модель Route

---

### 5. **ПРОБЛЕМА: Расстояние по прямой (Haversine)**

**Проблема:**
- Формула Хаверсина считает расстояние **по прямой** (как летит ворона)
- Реальное расстояние по дорогам может быть в 1.5-2 раза больше
- Нет учета рельефа, дорог, препятствий

**Последствия:**
- Расстояние занижено
- Время прохождения рассчитано неправильно
- Пользователи будут недовольны неточностью

**Решение:**
- Использовать OSRM API для расчета реального расстояния по дорогам
- Или добавить коэффициент (например, 1.5x для горных маршрутов)
- Кешировать результаты OSRM запросов

---

### 6. **ПРОБЛЕМА: beforeUpdate не всегда пересчитывает**

**Проблема:**
```javascript
if (data.places !== undefined) {
  await calculateRouteData(data);
} else {
  // Пересчет только если места меняются
}
```

**Последствия:**
- Если обновить только название маршрута, данные не пересчитаются
- Если координаты места изменились, маршрут не обновится автоматически
- Нужно вручную обновлять места, чтобы пересчитать

**Решение:**
- Всегда пересчитывать при обновлении маршрута
- Или добавить флаг `recalculate: true` для принудительного пересчета
- Слушать изменения координат мест через lifecycle hooks в Place

---

### 7. **ПРОБЛЕМА: Нет проверки на дубликаты**

**Проблема:**
- Можно добавить одно и то же место дважды в маршрут
- Нет валидации уникальности мест

**Последствия:**
- Маршрут может содержать дубликаты
- Расстояние будет рассчитано неправильно (включая дубликаты)

**Решение:**
- Добавить валидацию в lifecycle: проверять уникальность ID мест
- Удалять дубликаты перед расчетом

---

### 8. **ПРОБЛЕМА: Нет обработки ошибок БД**

**Проблема:**
- Если PostgreSQL недоступен, Strapi не запустится
- Нет fallback на SQLite для разработки

**Последствия:**
- Невозможно работать без PostgreSQL
- Сложная настройка для разработчиков

**Решение:**
- Добавить SQLite как fallback для разработки
- Или использовать Docker Compose для автоматической настройки PostgreSQL

---

### 9. **ПРОБЛЕМА: Нет кеширования расчетов**

**Проблема:**
- Каждый раз при сохранении маршрут пересчитывается заново
- Если места не менялись, расчет выполняется впустую

**Последствия:**
- Лишняя нагрузка на БД
- Медленная работа при частых обновлениях

**Решение:**
- Кешировать результаты расчета
- Пересчитывать только если изменились места или их координаты

---

### 10. **ПРОБЛЕМА: Нет связи с Go бэкендом**

**Проблема:**
- Strapi и Go бэкенд используют разные модели данных
- Strapi: many-to-many relation
- Go: RouteStop с order_num (промежуточная таблица)

**Последствия:**
- Два источника истины
- Синхронизация данных между Strapi и Go бэкендом невозможна
- Порядок мест теряется при экспорте из Strapi в Go

**Решение:**
- Использовать только Go бэкенд для создания маршрутов
- Или синхронизировать данные через API
- Или использовать Strapi только как админку, а Go - как основной API

## ⚠️ СРЕДНИЕ ПРОБЛЕМЫ

### 11. **Проблема: Нет валидации минимального количества мест**

- Можно создать маршрут с 0 или 1 местом
- Нет смысла в маршруте без мест

**Решение:** Добавить валидацию: минимум 2 места

---

### 12. **Проблема: Нет валидации типа маршрута**

- Можно выбрать несуществующий `type_id`
- Нет проверки существования Type перед сохранением

**Решение:** Добавить валидацию связей

---

### 13. **Проблема: Нет логирования**

- Ошибки только в console.error
- Нет структурированного логирования
- Нет мониторинга проблем

**Решение:** Добавить структурированное логирование

---

### 14. **Проблема: Нет тестов**

- Lifecycle hooks не покрыты тестами
- Нет проверки корректности расчетов

**Решение:** Написать unit-тесты для calculateRouteData

## 💡 РЕКОМЕНДАЦИИ ПО УЛУЧШЕНИЮ

### 1. Использовать промежуточную модель для порядка

Создать компонент `RouteStop`:
```json
{
  "place": { "type": "relation", "relation": "manyToOne", "target": "api::place.place" },
  "order_num": { "type": "integer", "required": true, "min": 0 }
}
```

И использовать его в Route:
```json
{
  "stops": {
    "type": "component",
    "repeatable": true,
    "component": "route.route-stop"
  }
}
```

### 2. Интеграция с OSRM

```javascript
const axios = require('axios');
const OSRM_URL = 'https://router.project-osrm.org/route/v1/driving';

async function getRealDistance(places) {
  const coordinates = places.map(p => `${p.longitude},${p.latitude}`).join(';');
  const response = await axios.get(`${OSRM_URL}/${coordinates}?overview=false`);
  return response.data.routes[0].distance / 1000; // в км
}
```

### 3. Динамическая скорость

```javascript
const routeType = await strapi.entityService.findOne('api::type.type', data.type_id);
const averageSpeed = routeType?.average_speed || 3; // км/ч
```

### 4. Валидация координат

```javascript
const placesWithoutCoords = sortedPlaces.filter(
  p => !p.latitude || !p.longitude
);
if (placesWithoutCoords.length > 0) {
  throw new Error(`У мест ${placesWithoutCoords.map(p => p.id).join(', ')} отсутствуют координаты`);
}
```
